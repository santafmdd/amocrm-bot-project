# AGENTS.md

## Project Goal

- Автоматизировать сбор отчетов из amoCRM.
- Последовательность этапов: сначала read-only capture, затем запись в таблицы, затем AI-анализ.
- Проект должен быть расширяемым: новые отчеты, новые страницы, новые фильтры без слома ядра.

## Current Scope

- Есть bootstrap-каркас проекта.
- Есть browser read-only MVP.
- Есть debug dumps для диагностики чтения UI.
- Есть config-driven каркас в `config/`.
- Полноценный UI управления пока не реализован.

## Safety Rules

- Не выполнять write/delete/save действия в amoCRM без отдельного явного задания.
- Не удалять файлы вне `project/`.
- Не трогать системные директории.
- Не делать массовые destructive actions.
- Не публиковать и не логировать чувствительные токены/cookies/storage state.

## Development Principles

- Не ломать рабочий MVP.
- Делать минимально инвазивные изменения.
- Не превращать один файл в монолит.
- Писать простой, понятный код.
- Добавлять type hints там, где это уместно.

## Architecture Conventions

- Browser/session слой отдельно.
- Page readers отдельно.
- Parsers отдельно.
- Report jobs отдельно.
- Config-driven слой отдельно.
- Новые источники (`analytics`, `deals`, `events`) добавлять как новые page/page-parser/report-profile, а не хардкодом в существующий reader.

## Config-Driven Rules

- Новые отчеты по возможности добавлять через YAML-конфиг.
- Новые теги/фильтры не должны хардкодиться в ядро.
- Target blocks и write modes должны жить в config.

## Documentation Rules

- При заметном изменении поведения обновлять `README.md`.
- При завершении шага обновлять `SESSION_HANDOFF.md`.
- Если меняется архитектурный слой, обновлять `ARCHITECTURE.md`.

## What Agents Must Not Do

- Не хардкодить текущие три тега как единственно возможные.
- Не предполагать, что все отчеты будут только из страницы "Аналитика".
- Не завязывать архитектуру только на один экран amoCRM.
- Не удалять debug-слой без явной причины.
- Не скрывать ошибки, если parser не сработал.
