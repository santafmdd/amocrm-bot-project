# Analytics Layout Rules

## 1) DSL в жёлтых строках

Одна жёлтая строка = один конфиг блока.

Формат:
- текст до первого `:` = человекочитаемое имя блока;
- после `:` = DSL фильтров.

Операторы/разделители:
- `;` = AND внутри одного сценария;
- `|` = несколько значений одного поля одновременно;
- `||` = альтернативные сценарии (выполняются отдельно);
- `=` = обычное/точное значение;
- `^=` = prefix match.

Примеры:
1. `Машэкспо: Даты=Созданы; Период=За все время; Воронка=Привлечение (2 месяца); Теги=машэкспо || Даты=Созданы; Период=За все время; Воронка=Привлечение (2 месяца); UTM Source^=conf_novosib_mechanical_engineering_2026`
2. `Все 3 выставки: Даты=Созданы; Период=За все время; Теги=машэкспо|инглегмаш-2026|уралстрой-2026`
3. `Общий срез: Даты=Созданы; Период=За все время; Воронка=Привлечение (2 месяца); Теги=машэкспо|инглегмаш-2026; UTM Source^=conf_`

## 2) Нормализация полей

Поддержанные normalized keys:
- `Теги` -> `tags`
- `UTM Source`, `utm source`, `utm_source` -> `utm_source`
- `Воронка` -> `pipeline`
- `Период` -> `period`
- `Даты` -> `dates_mode`

Важно:
- `raw_field_name` сохраняется всегда для прозрачных логов;
- мягкая нормализация без «телепатии»;
- никаких хардкодов под конкретные названия выставок.

## 3) Правило выбора лучшего сценария (`||`)

Сценарии ранжируются так:
1. больше `total_count`;
2. при равенстве — больше `non_empty_stage_rows`;
3. при полном равенстве — первый успешный сценарий.

## 4) Поиск блока на листе

Нельзя использовать fixed coordinates.

Нужно искать:
- жёлтую DSL-строку;
- под ней header row stage-таблицы;
- заголовки вида `Этап/Статус` + `Все/Активные/Закрытые` (допускаются alias-варианты).

Блоки могут двигаться по листу и менять порядок.

## 5) Правила записи

- не очищать весь лист;
- не ломать merge/цвета/форматирование;
- обновлять только числовые ячейки stage-блока;
- сопоставлять строки по stage name, а не по фиксированному номеру строки;
- stage в шаблоне без данных -> `0`;
- stage в данных без строки в шаблоне -> warning (без автодобавления строк).

## 6) Текущие ограничения

- нижняя таблица отказов игнорируется на этом этапе;
- prefix match `^=` выполняется в пределах текущих UI-возможностей (иначе — явный warning);
- full per-scenario UI re-run для `||` выделен в отдельный следующий шаг.
