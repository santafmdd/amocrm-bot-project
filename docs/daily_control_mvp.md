# Daily Control MVP (LLM-First)

## Scope

`Дневной контроль` active path is implemented as an LLM-first manager-day pipeline.

One row = one manager + one control day + one period.

## Architecture

Pipeline modules live in `src/deal_analyzer/daily_control/`:

1. `source_reader.py`: reads source rows from `Разбор звонков`.
2. `day_grouper.py`: groups rows by `period_start/period_end/date/manager`, computes counts and mixes.
3. `roks_oap_resolver.py`: picks current/previous ROKS OAP sheets by period end month.
4. `roks_oap_parser.py`: parses manager metrics from fixed block layout.
5. `daily_analyzer.py`: builds LLM context and requests strict JSON per manager-day.
6. `style/*`: optional post-processing and LLM rewrite guard.
7. `validation/*`: payload and text lint preflight.
8. `idempotency.py`: duplicate/conflict detection with key:
   - `period_start|period_end|control_day_date|manager_name|deals_count|calls_count`
9. `writer_plan.py`: plan-only layer for safe inserts/appends.
10. `sheets_writer.py`: Google Sheets write adapter (dry-run/live).
11. `artifacts.py`: artifact persistence.
12. `cli.py`: discover/build/write orchestration.

Legacy entrypoints:
- `src/deal_analyzer/daily_control_cli.py`
- `src/deal_analyzer/daily_control_builder.py`
- `src/deal_analyzer/daily_control_writer.py`

are wrappers over the package.

## LLM Contract

For each manager-day, analyzer sends full grouped context and expects strict JSON:

```json
{
  "date": "YYYY-MM-DD",
  "day_label": "понедельник",
  "manager_name": "Имя Фамилия",
  "department": "",
  "base_mix": "",
  "product_mix": "",
  "main_pattern": "",
  "strengths": "",
  "growth_zones": "",
  "why_it_matters": "",
  "what_to_fix": "",
  "what_to_tell_employee": "",
  "expected_effect_quantity": "",
  "expected_effect_quality": "",
  "score_0_100": 0,
  "criticality": "low|medium|high",
  "training_needed": false,
  "training_topic": "",
  "evidence_short": "",
  "data_limitations": ""
}
```

Runtime routing:
- main model: configurable (example: `gemma4:31b-cloud`)
- fallback model: configurable (example: `deepseek-v3.1:671b-cloud`)
- if both return invalid JSON: row is produced with explicit marker:
  - `не сформировано: llm_json_invalid`
  - backend: `deterministic_fallback`

No scripted hardcoded management phrases should be generated by code in active path.

## ROKS OAP Parsing

For period ending `2026-04-24`:
- current: `РОКС ОАП-апрель 2026`
- previous: `РОКС ОАП-март 2026`

Metric block layout is parsed by known offsets:
- Department block and manager blocks by row anchors.
- Month fact from column `D`.
- `Дозвоны` fallback: sum weekly facts from `F/H/J/L/N` when `D` is empty.

Snapshot artifact includes:
- selected sheets,
- block row anchors,
- parsed metrics,
- parse status and warnings.

## Safety and Idempotency

- Existing daily rows are read before write.
- Existing exact identity row -> skip.
- Same period/date/manager but different counts -> conflict.
- Conflicts are reported in artifact and do not overwrite by default.
- Writer plan is always generated before write.

## Commands

Discover:

```powershell
python -m src.deal_analyzer.daily_control.cli discover --config <config> --workbook "РОКС 2026" --daily-sheet "Дневной контроль"
```

Build dry-run:

```powershell
python -m src.deal_analyzer.daily_control.cli build --config <config> --period-start 2026-03-30 --period-end 2026-04-24 --source-sheet "Разбор звонков" --daily-sheet "Дневной контроль" --main-model gemma4:31b-cloud --fallback-model deepseek-v3.1:671b-cloud --dry-run
```

Write dry-run:

```powershell
python -m src.deal_analyzer.daily_control.cli write --config <config> --run-dir <run_dir> --daily-sheet "Дневной контроль" --dry-run --strict-preflight
```

Real write must be executed only by explicit operator command after artifact review.
